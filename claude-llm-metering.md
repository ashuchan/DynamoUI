# LLM Metering Feature ‚Äî Implementation Plan (v2)

## Overview

Add a generic metering subsystem that records every LLM interaction triggered anywhere in DynamoUI ‚Äî runtime query synthesis, scaffold pattern seeding, and any future caller. The layer is provider-level, not endpoint-level: a `MeteringLLMProvider` decorator wraps any underlying provider automatically, so new LLM callers are metered without touching the metering code.

---

## Internal DB Schema

All DynamoUI-managed tables live in a dedicated PostgreSQL schema `dynamoui_internal`, separate from the business data managed by adapters. This prevents naming collisions, simplifies permission grants, and makes it clear what DynamoUI owns vs what the application owns.

### Configuration

New setting added to `InternalSettings` (new Pydantic settings class):

```
DYNAMO_INTERNAL_SCHEMA   default: "dynamoui_internal"
DYNAMO_INTERNAL_DB_URL   optional ‚Äî defaults to the write pool URL if not set
```

The write pool user (`dynamoui_writer`) must have CREATE / INSERT / UPDATE / SELECT on `dynamoui_internal`. A setup SQL snippet will be included in the migration.

### Schema Definition Language (SDL)

The **canonical source of truth** for the internal schema is the SQLAlchemy `Table` definitions in `backend/metering/models/tables.py`. A human-readable DDL file is generated from these and committed at `backend/metering/schema/metering_schema.sql`. This file:
- Is regenerated by running `python -m backend.metering.schema.export`
- Is used for manual review, audit, and bootstrapping fresh environments
- Is checked in CI against the live SQLAlchemy metadata to detect drift

Alembic reads from the SQLAlchemy metadata (not the SQL file). The SQL file is derived from it.

### `metering_cost_rates`

Append-only pricing ledger. Each price change produces a new row ‚Äî existing rows are **never updated or deleted**. The table is its own audit trail.

| Column | Type | Notes |
|---|---|---|
| `id` | SERIAL PK | Referenced by `metering_llm_interactions.cost_rate_id` |
| `provider` | VARCHAR(64) NOT NULL | |
| `model` | VARCHAR(128) NOT NULL | |
| `input_cost_per_1k` | NUMERIC(10, 8) NOT NULL | USD per 1 000 input tokens |
| `output_cost_per_1k` | NUMERIC(10, 8) NOT NULL | USD per 1 000 output tokens |
| `thinking_cost_per_1k` | NUMERIC(10, 8) nullable | NULL if provider doesn't expose thinking tokens separately |
| `effective_from` | DATE NOT NULL | Rate applies from this date (inclusive) |
| `effective_to` | DATE nullable | NULL = currently active; set when superseded by a new row |
| `change_reason` | TEXT NOT NULL | Required. e.g. `"Anthropic April 2026 price reduction"` |
| `source_reference` | TEXT nullable | URL to official pricing announcement |
| `created_by` | VARCHAR(255) NOT NULL | Who inserted this row (operator identity or `"seed"`) |
| `created_at` | TIMESTAMPTZ NOT NULL | `DEFAULT NOW()` |

**Constraints**:
- UNIQUE `(provider, model, effective_from)`
- CHECK `(effective_to IS NULL OR effective_to >= effective_from)`
- No UPDATE or DELETE ‚Äî enforced at the DAO layer; a PostgreSQL rule can enforce this in production

**Mutations through `CostRateDAO` only** ‚Äî the DAO calls `supersede_active_rate()` which:
1. Sets `effective_to = today - 1` on the currently active row for the same (provider, model)
2. INSERTs the new row with `effective_from = today`
3. Both writes happen in a single transaction

Seed data (verify before production deploy):

```
anthropic / claude-haiku-4-5-20251001   input=0.00080  output=0.00400  thinking=0.00080  effective_from=2025-01-01
anthropic / claude-sonnet-4-6           input=0.00300  output=0.01500  thinking=0.00300  effective_from=2025-01-01
google    / gemini-1.5-flash            input=0.00007  output=0.00030  thinking=NULL      effective_from=2025-01-01
```

## Layer Responsibilities

### DTOs (`dto/`)

Pydantic v2 models. No business logic. Three shapes per entity:
- `*CreateDTO` ‚Äî fields required to write a new row
- `*UpdateDTO` ‚Äî fields that may be patched after creation (operation outcomes only)
- `*ReadDTO` ‚Äî full row shape returned from DAO reads (includes DB-generated fields: `id`, `created_at`)

DTOs are the contract between the service layer and the DAO layer. The API layer also uses `*ReadDTO` as its response schema.

```python
# Example
class LLMInteractionCreateDTO(BaseModel):
    operation_id: UUID
    tenant_id: UUID | None
    interaction_type: str
    provider: str
    model: str
    prompt_tokens: int
    completion_tokens: int
    thinking_tokens: int = 0
    thinking_summary: str | None
    latency_ms: int
    success: bool
    error_message: str | None = None
    # cost_usd and cost_rate_id are NOT in CreateDTO ‚Äî computed by CostCalculator before DAO insert
```

### DAOs (`dao/`)

Own all SQL. Accept DTOs, execute against the write pool, return `*ReadDTO` or raise.

```python
class OperationDAO(BaseDAO):
    async def insert(self, dto: OperationCreateDTO) -> OperationReadDTO: ...
    async def update_outcome(self, operation_id: UUID, dto: OperationUpdateDTO) -> None: ...

class InteractionDAO(BaseDAO):
    async def insert(self, dto: LLMInteractionCreateDTO, cost_usd: Decimal, cost_rate_id: int) -> LLMInteractionReadDTO: ...

class CostRateDAO(BaseDAO):
    async def get_active_rate(self, provider: str, model: str, on_date: date) -> CostRateReadDTO | None: ...
    async def supersede_active_rate(self, dto: CostRateCreateDTO) -> CostRateReadDTO: ...
    async def list_rates(self, provider: str | None, model: str | None) -> list[CostRateReadDTO]: ...
```

`BaseDAO` holds the write pool reference and `schema_name` string so all child DAOs prepend `dynamoui_internal.` automatically via the SQLAlchemy `Table(schema=...)` definition.

### CostCalculator (`cost.py`)

Stateless. Takes a `CostRateReadDTO` and token counts, returns `Decimal`:

```python
class CostCalculator:
    def compute(
        self,
        rate: CostRateReadDTO,
        prompt_tokens: int,
        completion_tokens: int,
        thinking_tokens: int,
    ) -> Decimal:
        ...
```

The `MeteringService` resolves the active rate via `CostRateDAO.get_active_rate()` at interaction record time, then passes both to `CostCalculator.compute()`. The resolved `cost_rate_id` is stored on the interaction row.

### MeteringContext (`context.py`)

Python `contextvars.ContextVar` carries the active metering context through async call chains without threading it through every function signature:

```python
@dataclass
class MeteringContext:
    operation_id: UUID
    interaction_type: str       # "query_synthesis" | "pattern_seeding" | ...
    tenant_id: UUID | None = None

_current_metering: ContextVar[MeteringContext | None] = ContextVar(
    "_current_metering", default=None
)

def set_metering_context(ctx: MeteringContext) -> Token: ...
def get_metering_context() -> MeteringContext | None: ...
def clear_metering_context(token: Token) -> None: ...
```

### MeteringLLMProvider (`provider_decorator.py`)

Wraps **any** `LLMProvider`. This is the only integration point for all current and future LLM callers. The decorated provider automatically meters without the caller knowing:

```python
class MeteringLLMProvider(LLMProvider):
    def __init__(
        self,
        inner: LLMProvider,
        metering_service: MeteringService,
        provider_name: str,
        model: str,
    ) -> None: ...

    async def complete(self, system_prompt: str, user_prompt: str) -> str:
        ctx = get_metering_context()
        start = time.monotonic()
        try:
            result = await self._inner.complete(system_prompt, user_prompt)
            # parse token usage from raw response (requires provider to expose it ‚Äî see below)
            ...record interaction via metering_service...
            return result
        except Exception:
            ...record failed interaction...
            raise
```

Because the existing `LLMProvider.complete()` returns only a string, the provider implementations need to be changed to return a richer type internally. See "Provider Changes" below.

### MeteringService (`service.py`)

High-level API. Used by callers (`/resolve`, scaffolder). Wraps all DAO calls in try/except so metering failures never propagate:

```python
class MeteringService:
    async def start_operation(
        self,
        operation_type: str,
        tenant_id: UUID | None = None,
        session_id: UUID | None = None,
        user_id: str | None = None,
        ip_address: str | None = None,
        metadata: dict | None = None,
    ) -> UUID:
        """Insert a metering_operations row. Returns operation_id."""

    async def complete_operation(
        self,
        operation_id: UUID,
        dto: OperationUpdateDTO,
    ) -> None:
        """Update operation row with outcome. Always call in a finally block."""

    async def record_llm_interaction(
        self,
        dto: LLMInteractionCreateDTO,
    ) -> None:
        """
        Called by MeteringLLMProvider only.
        Resolves cost rate, computes cost, inserts interaction row.
        """
```

## Generic Integration ‚Äî How Callers Plug In

All callers follow the same three-line pattern. No metering code inside business logic beyond this:

```python
# 1. Start an operation (creates metering_operations row)
operation_id = await metering_service.start_operation(
    operation_type="resolve",
    ip_address=client_ip,
)

# 2. Set context var so MeteringLLMProvider can read operation_id without being passed it
token = set_metering_context(MeteringContext(
    operation_id=operation_id,
    interaction_type="query_synthesis",
))

try:
    result = await existing_business_logic()
finally:
    clear_metering_context(token)
    await metering_service.complete_operation(operation_id, OperationUpdateDTO(...))
```

The `MeteringLLMProvider` reads the context var automatically ‚Äî no changes to `QuerySynthesiser`, `PatternSeeder`, or any future LLM caller beyond wiring at startup.

### Current LLM call sites and their `interaction_type`:

| File | Call site | `interaction_type` |
|---|---|---|
| `skill_registry/llm/query_synthesiser.py` | `_synthesiser.synthesise()` | `query_synthesis` |
| `skill_registry/llm/pattern_seeder.py` | `seed_patterns()`, `seed_patterns_batch()` | `pattern_seeding` |
| Future | anything using `LLMProvider` | add a new string constant |

## Provider Changes

The `LLMProvider.complete()` signature must return a richer type so the `MeteringLLMProvider` decorator can access token counts without a second API call. The base class contract changes:

```python
@dataclass
class LLMRawResponse:
    text: str
    prompt_tokens: int
    completion_tokens: int
    thinking_tokens: int = 0
    thinking_summary: str | None = None  # first 500 chars of thinking block
    model: str = ""                       # model name as echoed by provider

class LLMProvider(ABC):
    @abstractmethod
    async def complete(self, system_prompt: str, user_prompt: str) -> LLMRawResponse:
        """Return a rich response; text field is the generated content."""
```

External callers that currently do `result = await provider.complete(...)` and treat `result` as a string change to `result.text`. The `MeteringLLMProvider` wraps the inner provider, intercepts `LLMRawResponse`, records the interaction, then returns the same `LLMRawResponse` unchanged ‚Äî the caller sees no difference.

`_NullProvider` returns a zero-token `LLMRawResponse(text="")`.

## Alembic Setup (first real use)

This feature bootstraps the Alembic migration system. All metering tables are created in the `dynamoui_internal` schema.

**Setup steps:**
1. `alembic init alembic/` at repo root
2. Configure `alembic/env.py`:
   - Load `InternalSettings` from env
   - Use the write pool URL
   - Set `include_schemas=True`, `version_table_schema="dynamoui_internal"`
   - Point `target_metadata` at `metering_metadata` from `backend/metering/models/tables.py`
3. First migration: `alembic revision -m "create_dynamoui_internal_schema_and_metering_tables"`
   - Manually written (not autogenerate) for precision
   - Creates `dynamoui_internal` schema if not exists
   - Creates all 4 tables with indexes and constraints
   - Seeds `metering_cost_rates` with initial rates
4. `alembic upgrade head` on deploy

**Migration naming convention**: `YYYYMMDD_NNN_description.py` ‚Äî date prefix makes ordering unambiguous.

## New API Endpoints (`/metering/*`)

Read-only. No auth in Phase 1 (follows Phase 2 auth work).

| Endpoint | Description |
|---|---|
| `GET /metering/summary?from=&to=` | Totals: operations, LLM calls, tokens, cost ‚Äî optional `operation_type` filter |
| `GET /metering/operations?page=&page_size=&operation_type=` | Paginated `OperationReadDTO` list |
| `GET /metering/operations/{id}` | Single operation + its `LLMInteractionReadDTO[]` |
| `GET /metering/cost-by-model?from=&to=` | Aggregated cost by `(provider, model)` |
| `GET /metering/cost-rates` | All `CostRateReadDTO` rows (full history) |
| `POST /metering/cost-rates` | Add a new rate version ‚Äî accepts `CostRateCreateDTO` |

The `POST /metering/cost-rates` endpoint is the only write path for rate management. It calls `CostRateDAO.supersede_active_rate()` in a transaction. In Phase 2 this will require an admin role JWT claim.

## Cost Rate Versioning & Audit

The design is append-only ‚Äî the table IS the audit trail:

- Every price change is a new row; no row is ever updated or deleted
- The DAO enforces this: `CostRateDAO` has no `update()` or `delete()` methods
- A PostgreSQL rule (`CREATE RULE no_update_cost_rates ...`) can enforce this at DB level for additional safety
- `created_by` and `change_reason` are required ‚Äî the DAO raises `ValueError` if either is missing
- `metering_llm_interactions.cost_rate_id` is a FK to the exact row used ‚Äî you can always recompute any historical cost by joining on this FK

**To update a rate:**
1. Call `POST /metering/cost-rates` with the new `CostRateCreateDTO`
2. DAO wraps in a transaction: updates `effective_to` on the old active row, inserts the new row
3. All future interactions use the new rate ID; all past interactions still reference the old rate ID

## Multi-Tenancy Evolution Path

**Phase 1 (now)**: `tenant_id` nullable everywhere. `MeteringContext.tenant_id = None`.

**Phase 2 (multi-tenant)**:
1. Add `tenants` table to `dynamoui_internal` (id UUID, name, slug)
2. Seed a `default` tenant
3. Alembic migration: backfill `tenant_id`, make NOT NULL, add FK constraints
4. Add PostgreSQL RLS policies on `metering_operations` and `metering_llm_interactions`
5. Wire `tenant_id` from JWT claims into `MeteringContext` at the endpoint level

No changes to `MeteringService`, `MeteringLLMProvider`, or DAOs beyond the FK becoming required.

## Resilience

- All metering writes use the **write pool** (`DYNAMO_INTERNAL_DB_URL` or write URL)
- `MeteringService` wraps every DAO call in `try/except Exception` ‚Äî metering failures log at WARN and are swallowed
- `complete_operation()` is called in a `finally` block in every caller
- `asyncio.shield(complete_operation(...))` protects the final write from request cancellation
- If the DB is unavailable at startup, `MeteringService` starts in a no-op mode (all methods return immediately), logging one WARN at boot

## Out of Scope for This Phase

- Auth on `/metering/*` endpoints
- Real-time streaming / webhooks for metering events
- Budget alerts or quota enforcement
- Per-tenant dashboards
- Retroactive cost recalculation when rates change (can be done via a SQL query using `cost_rate_id` FK ‚Äî no code needed)
